# Quest Structure & Scenario Engine

## 1. 概要 (Overview)
本仕様書は、クエストのデータ構造と出現ロジック、シナリオエンジンのフロー、およびインベントリ・裏切りメカニクスを統合定義する。

---

## 2. データ構造

### 2.1 scenarios テーブルとID管理ルール

```sql
CREATE TABLE scenarios (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  slug TEXT UNIQUE,
  title TEXT NOT NULL,
  quest_type TEXT DEFAULT 'normal',   -- 'normal' | 'special' | 'main'
  location_id UUID REFERENCES locations(id),
  difficulty INT DEFAULT 1,
  rewards JSONB,                       -- { exp: number, gold: number, items: [...], reputation: number, etc. }
  conditions JSONB,                    -- 受注条件（名声などの制限はここに一元化）
  days_success INT DEFAULT 1,
  days_failure INT DEFAULT 1,
  flow JSONB,                          -- シナリオノードの木構造
  short_description TEXT,              -- クエスト一覧ボード用テキスト (最大40文字程度)
  full_description TEXT,               -- 詳細モーダル用長文フレーバーテキスト
  status TEXT DEFAULT 'published'
);
```

**【IDの運用ルール】**
*   **公式（System）**: `5001`, `6001`, `7001` 等の整数ベースのIDを使用する。データベースの制約上UUIDカプセル化が必要な場合はスクリプトで決定的なUUIDへの変換を行うか、スキーマをTEXTに変更する運用を行うが、CSVなど人間の管理レイヤーにおいては「整数」を正当なIDとして取り扱う。
*   **UGC（User Generated）**: 将来のプレイヤー作成クエストなどは重複回避のため完全なUUID（`gen_random_uuid()`）を用い、公式クエストと安全に混在・共存させる。

### 2.2 items / inventory テーブルの構造

```sql
CREATE TABLE items (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  slug TEXT UNIQUE,
  name TEXT NOT NULL,
  type TEXT DEFAULT 'consumable',  -- 'consumable', 'trade_good', 'skill', 'key_item'
  base_price INT DEFAULT 100,
  effect_data JSONB,
  is_skill BOOLEAN DEFAULT FALSE,  -- trueの場合スキルとしてデッキ装備可能
  description TEXT
);

CREATE TABLE inventory (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id),
  item_id BIGINT REFERENCES items(id),
  quantity INT DEFAULT 1,
  is_equipped BOOLEAN DEFAULT FALSE,
  is_ugc BOOLEAN DEFAULT FALSE
);
```

## 3. クエスト出現ロジック (Appearance Logic) と受注条件

クエストの表示・受注制限は `conditions` JSONB に集約される（`api/location/quests` でフィルタリングを行う）。

**`conditions` で指定可能な主なキー:**
*   `min_reputation` (Number): プレイヤーの名声がこの値**以上**でなければ受注不可。（例: `80` = 英雄向け）
*   `max_reputation` (Number): プレイヤーの名声がこの値**以下**でなければ受注不可。（例: `-80` = 悪党向け）
*   `completed_quest` (String): 前提となるクエストの `slug`（あるいはID）をクリア済みか。
*   `location_tags` (Array): 出現拠点フラグ。

> **⚠ 重要**: 過去存在した単独カラムとしての `min_reputation`, `max_reputation` 等は廃止（または無視）し、すべて `conditions` JSONB内で数値ベースで管理・判定する。

## 4. シナリオノード構造 (flow JSONB)

`flow` カラムにはツリー構造のノードデータが格納される。

### 4.1 公式定義ノード（`type`）一覧

| Type | 機能・役割 |
|---|---|
| `text` | テキストの表示と文字送り。 |
| `choice` | 選択肢の提示。ユーザーの回答により `next` へ分岐する。 |
| `battle` | バトルの開始（`enemy_slug` またはグループIDを指定）。 |
| `shop` | クエスト中専用ショップを開く。 |
| `travel` | ワールドマップを開き、移動先を指定させる。 |
| `camp` | 野営地メニュー（HP回復、デッキ調整など）。 |
| `reward` | クエストの途中報酬（アイテム等）を付与する。 |
| `end` | クエスト成功（または失敗）として完了処理へ移行。 |
| **`check_delivery`** | 【追加】指定アイテム（id,個数）を所持しているか判定し、**消費（没収）**して進む。 |
| **`check_possession`** | 【追加】指定アイテムや固有フラグを判定するが、**消費・変更しない**。 |
| **`join` / `leave`** | 【追加】ゲストNPCのパーティ加入・離脱。(`is_escort_target`: true で死亡時即失敗) |
| **`trap` / `modify_state`**| 【追加】テキストと共に、HP割合低下や手札制限などの環境デバフをクエストステートに直接課す。 |
| **`modify_flag`** | 【追加】クエスト進行中のみ有効な内部変数（呪いのカウントや正解数など）を加算/減算する。 |
| **`modify_reputation`**| 【追加】道中の選択やイベントにより、名声（アライメント）を即座に増減させる。 |

## 5. 裏切りシステム (Betrayal Mechanic) と特使アイテム

### 5.1 基本仕様
クエスト中に取得または要求された「キーアイテム・消耗品」などを、クエストクリア前にショップに売却すると **「裏切り」** と判定される。
*   **ペナルティ**: 警告の後、承諾するとクエストは即座に失敗扱いとなり、名声の低下やクエスト失敗ペナルティが課される。

### 5.2 スポットシナリオ専用スキルの例外（売却不可・裏切り免除）
「四神の宝珠」や「五英霊の誓約」など、**クエストのボス攻略のためにデッキに装備して発動する固有アイテム（スキル）**は以下の例外扱いとし、裏切りの対象としない。
*   データベース上は `is_skill = true`, `type = 'skill'` として定義される。
*   システム的に**「ショップ売却不可」の専属スキル**として扱うため、プレイヤーが誤って売却してペナルティを受ける事態を防ぐ。

## 6. クエスト完了処理 & 履歴管理 (Quest Complete)
1. **加齢・成長処理**: 経過日数に基づく年齢計算。
2. **報酬付与**: `quest.rewards` の金額・EXP・アイテム・最終名声変動を適用。
3. **クリア履歴の保存**: `user_completed_quests` テーブルに記録。
4. **ステートリセット**: ゲストNPCの離脱・内部フラグ（呪い・正解数など）の完全破棄。
