# spec_new_onboarding: 新規オンボーディングフローおよびおまかせデッキ編成仕様書

**Version**: 4.0  
**Date**: 2026-06-25  
**Status**: Approved  

---

## 1. 概要

新規ユーザーの初期離脱率低下およびデータ損失防止を目的とし、既存のオンボーディングおよびチュートリアル（宿屋での `TutorialModal`）を完全に廃止し、世界観説明 ➔ パック開封 ➔ おまかせデッキ編成 ➔ 第1話「始まりの轍」（シナリオID: `6001`）の強制開始までを一本道で繋ぐ「レール式オンボーディング」を導入する。  
また、第1話クリア後にはユーザーの属性（ゲスト / 本登録）に合わせたプロモーション表示を行い、その後のゲームサイクル（酒場雇用 ➔ ギルド受注 ➔ マップ出発）をステップバイステップでナビゲートする。

---

## 2. 開発の分離・競合防止方針

並行して実施されている可能性のある他の機能開発やクエスト改修とコードが競合するのを防ぐため、本開発は**完全な独立分離設計**で実装する。

1. **新規モジュール・エンドポイントの分離**:
   - チュートリアル用のパック開封画面は、既存の `AcademyModal.tsx` を一切改変せず、新規に作成する `OnboardingAcademyModal.tsx` で完結させる。
   - おまかせデッキ編成処理は、汎用のインベントリAPIを拡張するのではなく、専用エンドポイント `/api/inventory/auto-deck` を新設して処理する。
2. **コア処理エンジンへの介入排除**:
   - シナリオ処理やバトルなどの共通コアエンジン（`ScenarioEngine.tsx` や `BattleView.tsx`）のコードには一切手を加えず、画面遷移（Router）と宿屋画面（`/inn`）への帰還時のフラグ判定のみで本オンボーディングフローを完結させる。これにより、他のクエスト開発との競合影響を「ゼロ」とする。

---

## 3. 開発ルール (KI) への準拠

開発ガイドライン `AGENTS.md` の制約に基づき、以下の実装設計を適用する。

1. **アトミックなキー付与と重複防止 (ルール 12.3 準拠)**:
   - 新規登録時に「知識と契約の鍵 (ID: 76)」を付与する際、単なる `insert` ではなく `ON CONFLICT (user_id, item_id) DO UPDATE SET quantity = inventory.quantity + EXCLUDED.quantity` による upsert 制御を用いて安全に実行する。
2. **非同期通信時の二重発火・連打防止ガード (ルール 3.1 & 10.6 準拠)**:
   - `OnboardingAcademyModal.tsx` で「おまかせデッキ編成 ＆ 冒険を開始」ボタンがクリックされた際、非同期通信（自動編成 ➔ 受注 ➔ チュートリアル完了 ➔ 遷移）の処理中は、UI側でローカルステート `isProcessing` を `true` に設定し、ボタンを無効化（`disabled`）して二重送信を物理的にブロックする。
3. **限定パッケージの二重購入防止 (ルール 12.2 準拠)**:
   - `StarterPackPromoModal` を表示する際、フロントエンドで `userProfile.has_purchased_starter` および `userProfile.has_purchased_elite` を参照する。
   - 両方のパックを既に購入済みの場合は、モーダル自体の表示をスキップ（表示しない）する。
   - モーダル内部では、購入済みのパックを非活性化（「購入済み」と表示）する。
4. **決済パラメータキーの整合 (ルール 13.1 準拠)**:
   - 課金モーダルを起動する際は、Stripe Webhook 等で使用されている正確なパッケージキー `starter_pack` および `elite_pack` を渡して決済プロセスを実行する。

## 4. オンボーディング専用モーダル (`OnboardingAcademyModal`)

新規モーダル `src/components/inn/OnboardingAcademyModal.tsx` を実装する。  
本モーダルはチュートリアル未完了（`userProfile.is_tutorial_completed === false`）の際に自動表示され、完了するまで閉じる（Xボタンなど）ことはできない。

### 4.1 各フェーズの遷移

#### フェーズ 1: 世界観説明
- **UI内容**: ダークファンタジー調で装飾されたカードUI上に、世界観を伴う以下の説明テキストを表示する。
  > 「目覚めよ、名もなき旅人よ。  
  > 混沌と秩序が交錯するこの地を歩むには、運命を切り拓く力……すなわち『カード』の契約が必要となる。  
  > 学舎の門を叩き、古の契約を結ぶのだ。」
- **操作**: 「魔術学院へ進む」ボタンのみを表示。

#### フェーズ 2: チュートリアル用パック開封
- **UI内容**: 「黎明の知識と古の契約（`basic`）」パックの画像を表示。
- **制限**: ゴールドでの購入ボタンは完全に排除し、ユーザー作成時に付与される「知識と契約の鍵（`item_basic_key`）」でのみ引けるように制限。

#### フェーズ 3: 開封演出・カードめくり
- **UI内容**: 既存の `AcademyModal` と同様の引き裂きおよび5枚のカード表面めくり演出を表示。

#### フェーズ 4: デッキ自動編成・第1話開始
- **UI内容**: 開封完了後、「もう一度引く」「閉じる」ボタンを非表示にし、代わりに「おまかせデッキ編成 ＆ 冒険を開始」ボタンを配置。
- **制限**: 処理実行中はボタンを `disabled` に制御して連打によるAPI多重送信を防ぐ。
- **ボタン押下時の動作**:
  1. `POST /api/inventory/auto-deck` を呼び出し、所持カードから自動デッキ編成を実行。
  2. `POST /api/quest/start` を呼び出し、第1話「始まりの轍」（ID: `6001`）を受注。
  3. `POST /api/profile/complete-tutorial` を呼び出し、チュートリアルを完了。
  4. 完了後、自動的に `/quest/6001` へ画面遷移。

## 5. バックエンド API 仕様

### 5.1 `POST /api/inventory/auto-deck` (おまかせデッキ編成)
- **認証**: JWTトークンによるユーザー特定。
- **処理ロジック**:
  1. ユーザーの `max_deck_cost` を `user_profiles` から取得（デフォルト: 10）。
  2. `user_skills`（正規スキル）および `inventory`（レガシースキル）の所持カードをすべて取得。
  3. 一旦すべての装備フラグ（`is_equipped`）を `false` にリセット。
  4. カードを `deck_cost`（コスト）の降順にソート。
  5. 降順に走査し、`累計コスト + 当該カードコスト <= max_deck_cost` である場合、当該カードの `is_equipped` を `true` に設定（貪欲法）。
  6. 設定した装備状態をデータベースへ保存。
  7. 装備した正規スキルの `card_id`（最大6枚）を抽出し、`user_profiles.signature_deck` を更新。
  8. 成功時に `{ success: true, current_deck_cost }` を返却。

### 5.2 `POST /api/profile/init` (初期鍵の付与)
- **処理ロジック**:
  - 新規プロフィール登録（データベースへのINSERT）が行われる際、自動的に `inventory` テーブルに「知識と契約の鍵（`item_id`: 76, 数量: 1, `is_equipped`: false）」をインサートする。
  - SQL レベルで衝突を避けるため、**ON CONFLICT (user_id, item_id) DO UPDATE SET quantity = inventory.quantity + EXCLUDED.quantity** の upsert ロジックを使用する。

## 6. クリア後の属性別プロモーション

第1話クリア後に宿屋に戻った際、ユーザー種別に応じた案内を行う。

### 6.1 ゲストユーザー向け：本登録案内
- **トリガー**: 第1話（または他クエスト）クリア後、ユーザーがゲスト（`is_anonymous === true`）である場合。
- **表示内容**:
  - 新規生成アセット `promo_guest_register.png` をヘッダーに配置。
  - 世界観に沿った以下の案内メッセージを表示：
    > 「旅人よ、素晴らしい活躍であった。だが、忘れてはならない。  
    > 署名なき冒険譚は風に消えゆく宿命にある。  
    > お前の歩んだ確かな軌跡を、この世界の歴史書に深く刻み込むのだ。  
    > さすれば、お前の冒険の記録は永遠に失われることはないだろう……」
  - 「Googleアカウントで冒険の記録を残す（本登録）」ボタンをタップすると、既存の Google アカウント連携（`linkIdentity`）処理を実行。

### 6.2 本登録完了直後の連携遷移（ゲスト ➔ 本登録）
- **動作フロー**:
  1. ゲストユーザーが本登録（OAuth連携）を行う前に、`sessionStorage.setItem('wirth_dawn_just_registered', 'true')` を設定する。
  2. Google等の外部連携手続きを経て宿屋（`/inn`）へリダイレクトバックされる。
  3. `sessionStorage` に `'wirth_dawn_just_registered'` が存在し、かつ `userProfile.is_anonymous` が `false`（本登録ユーザー）に切り替わっていることを検知。
  4. 検知後、セッションフラグをクリアした上で、**自動的にスターター/エリートパック案内モーダル (`StarterPackPromoModal`) をポップアップ起動する**。
- **目的**: ゲスト登録を完了したユーザーを離脱させず、そのままプレミアムパックのプロモーションにシームレスに繋ぐ。

### 6.3 本登録ユーザー向け：スターター/エリートパック案内モーダル (`StarterPackPromoModal`)
- **トリガー**: 
  - ① 第1話クリア後の帰還時（すでに本登録済みの場合）。
  - ② ゲストユーザーが本登録を完了して宿屋にリダイレクトバックされた直後（上記 6.2 のフロー）。
  - ③ 本登録済みかつ第1話クリア済みのユーザーが「次回ログイン」した時（`sessionStorage` でセッション中1回のみポップアップするよう制御）。
- **表示内容**:
  - 新規生成アセット `promo_starter_pack.png` を配置。
  - **購入状態のチェック**: `userProfile.has_purchased_starter` / `has_purchased_elite` を参照し、購入済みのものは「購入済み」と非活性表示する（双方購入済みの場合はモーダル表示自体をスキップ）。
  - **世界観に沿った案内メッセージ**:
    > 「見事であった。  
    > 険しさを増すこれからの旅路に備え、旅人の力となる『スターターパック』、そして深淵なる知識を秘めた『エリートパック』に興味はあるか？興味があるのであれば、内容を確認し、汝のデッキをさらに強固なものとするが良い……」
  - **ボタンの選択肢と遷移**:
    * **「内容を確認する」ボタン**: タップ時にモーダルを閉じ、既存の課金画面（`BillingModal`）を `packageKey` ('starter_pack' または 'elite_pack') を添えて起動する。
    * **「すぐに冒険を開始する」ボタン**: タップ時にモーダルを閉じ、次のステップ（酒場での仲間雇用ナビ）に進む。

## 7. 冒険の旅立ちナビゲーション (目的ガイダンスバナー)

第1話クリア後、プレイヤーが段階的に拠点の全貌と主要なシステムを理解できるよう、ガイダンスバナーとハイライトによる同調的なステップナビゲーションを以下の順序で提供する。
この案内ツアーの進行状況は、`localStorage` の `'wirth_dawn_onboarding_tour_step'` （値: 1〜6、または完了時は `'completed'`）で一元管理される。

## ツアー中の共通動作仕様
- **他ボタンの非活性化**: ツアー進行中（ステップ 1〜6）、現在フォーカスされている推奨ボタン以外のすべての施設ボタン、ヘッダーボタン、およびマップボタンのタップ（クリック）を非アクティブ（無効化）とする。
- **同期的ステップ進行**: 推奨ボタンがタップされた際、実際の施設モーダルや設定画面を開くのではなく、同調的にステップを次に進める（`wirth_dawn_onboarding_tour_step` をインクリメントする）のみとする。これにより、不要なローディング時間や遷移待ちを根絶する。
- **完了と遷移**: 最後のステップ 6 （ワールドマップ）のコンパスボタンがタップされた際に、ツアーを `'completed'` とし、通常通りワールドマップへ画面遷移させる。

## 案内ステップ順序とテキスト
1. **① 宿屋/酒場**:
   - 推奨ボタン: `inn` (宿屋/酒場) 施設ボタンのみアクティブ。
   - バナーテキスト: 「💡 旅の目的: 宿屋/酒場では仲間を探すことができます。」
2. **② ギルド**:
   - 推奨ボタン: `guild` (ギルド) 施設ボタンのみアクティブ.
   - バナーテキスト: 「💡 旅の目的: ギルドでは様々なクエストを受けることができます。」
3. **③ 道具屋**:
   - 推奨ボタン: `shop` (道具屋) 施設ボタンのみアクティブ。
   - バナーテキスト: 「💡 旅の目的: 道具屋では冒険に役立つ様々なアイテムを購入することができます。」
4. **④ 魔術学院**:
   - 推奨ボタン: `magicAcademy` (魔術学院) 施設ボタンのみアクティブ。
   - バナーテキスト: 「💡 旅の目的: 魔術学院では魔導の契約（カードパック購入）やデッキの確認ができます。」
5. **⑤ 設定/課金**:
   - 推奨ボタン: ヘッダーの `設定 (Cog)` または `課金 (CreditCard)` ボタンのみアクティブ。
   - バナーテキスト: 「💡 旅の目的: 設定や課金（魔導ショップ）では、音量の調整、アカウントの管理、特別なパッケージの購入ができます。」
6. **⑥ ワールドマップ（出発）**:
   - 推奨ボタン: メインビジュアルエリアの `ワールドマップを開く` (Compass) ボタンのみアクティブ。
   - アピール演出: 金色（アンバー/ゴールド）のバウンス発光、右上バッジを「出発」に変更。
   - バナーテキスト: 「💡 旅の目的: 「出発する」を押して、ワールドマップから冒険に旅立ちましょう！」

## 8. 号外システムの一時廃止

ユーザー体験向上のため、以下の号外新聞シェア機能および画像生成処理を停止する。
- **UI側**: `QuestResultModal.tsx` 下部の「号外！冒険を共有しよう」セクション（SNSシェア機能）をレンダリングから除外。
- **API側**: `/api/quest/complete` および `/api/character/retire`, `/api/profile/init` 内の `share_data_list` の生成処理を抑止し、常に空の配列（`[]`）を返す。

## 9. プレビューテストプレイにおける不具合修正（Version 1.8 追記）

プレビュー環境のモバイルテストプレイで発生した不具合について、以下の修正を適用する。

1. **オンボーディングパックの演出追加**:
   - `OnboardingAcademyModal.tsx` のパックビジュアルに通常版と同様のホバー・タップ演出クラス `group hover:scale-105 active:scale-95 transition-all duration-300` および `animate-bounce` による「TAP TO OPEN」表示を実装し、引き心地を統一する。
2. **メインクエスト1話のノードデータ異常の解消**:
   - `page.tsx` (`/quest/[id]/page.tsx`) マウント時に Zustand から前回の `currentNodeId` を読み込む際、必ず保存されている `questId` と現在開いている `id` が一致している場合のみ復元するように修正し、ID不一致時は初期値からクエストを開始させる。
3. **本登録案内ポップアップ画像の配置**:
   - 0バイトとなっていた `public/images/promos/promo_guest_register.png` および未配置の `promo_starter_pack.png` に、生成済みの正常な画像ファイルをコピー配置する。
4. **本登録・Google連携完了時のリダイレクト競合の解消**:
   - `useInnPageState.ts` の OAuth コールバック検出時に、URL の `code` パラメータを即時に削除せず、`supabase.auth.onAuthStateChange` イベントでセッション更新（トークン交換完了）を検知した後にクリーンアップを行うよう変更し、連携後のアカウント移行（`is_anonymous: false`）を確実に反映する。
5. **案内バナーテキストの見切れ解消**:
   - `page.tsx` (`/inn/page.tsx`) のガイダンスバナーテキスト内の `truncate` クラスを削除し、モバイル環境でも折り返しによる全文表示を可能にする。

## 10. プレビューテストプレイの再バグ修正およびデータプリフェッチ仕様（Version 1.9 追記）

1. **オンボーディングガチャ画面の引き裂きスライド演出（ripping）の完全再現**:
   - `OnboardingAcademyModal.tsx` において、パックを引いた直後（API通信成功時）に直接めくり画面（`flip`）に移行していた箇所を変更。
   - 通常の魔術学院と同様に `'ripping'` ステップを新設し、「撕」「裂」の漢字が上下にスライドアウトするアニメーション演出（`renderRippingPack`）を800ms再生したのちにめくり画面へ遷移させる。
2. **オンボーディング第1話中の設定アイコン非表示化**:
   - 第1話受注完了時にチュートリアル完了フラグ（`is_tutorial_completed`）が `true` に切り替わってしまうため、`is_tutorial_completed === false` の条件チェックでは設定メニューが非表示にならない問題を解消する。
   - `QuestHeader.tsx` において、シンプルに `questId === '6001'`（第1話進行中）のときは常に設定アイコンボタンを非表示にし、ギブアップや途中離脱を完全に制限する。
3. **Google連携本登録完了時のセッション移行失敗の解消**:
   - `authToken.ts` でアクセストークン（JWT）がメモリ上にキャッシュされており、Googleログイン成功で Supabase の認証セッションが更新されたにもかかわらず、その後のデータベース同期やプロフィール取得で古い匿名のキャッシュトークンが送信され続けていた競合原因を特定。
   - `useInnPageState.ts` 内のOAuthコールバック処理（`handleOAuthCallback`）にて、手動での PKCE コードセッション交換（`exchangeCodeForSession`）の実行前後に `clearAuthTokenCache()` を呼び出し、メモリキャッシュを強制破棄・リフレッシュさせ、本登録完了状態（`is_anonymous: false`）へのデータベース同期を確実に成功させる。
4. **チュートリアル完了後の拠点ロード軽量化（プリフェッチキャッシュ）**:
   - 酒場、道具屋、クエストボードをロード時間なし（瞬時）で開けるようにするため、以下のプリフェッチとキャッシュ機構を導入する。
   - **道具屋のSWRキャッシュ化**: Zustandの `useGameStore` に道具屋データキャッシュ（`shopCache`）とAPIフェッチアクション `fetchShop` を追加し、`ShopModal.tsx` を酒場と同様のSWRパターン（60秒以内のキャッシュがあればロードなしで即表示、なければバックグラウンドフェッチ）に書き換える。
   - **第1話進行中のプリフェッチ実行**: 第1話「始まりの轍」（`/quest/6001`）の進行中に、非同期バックグラウンド処理として `/api/init-page?prefetch_quest_id=6001`（酒場・クエスト・噂話等の一括取得API）および `fetchShop()` をプリフェッチ実行し、Zustandストアにあらかじめデータをキャッシュしておく。これにより、あらかじめ第1話クリア想定（第2話が出現している状態）のキャッシュが構築される。

## 18. モーダルタップ領域ズレ、設定・課金アイコン消失、およびテストプレイ時のチュートリアル未開始バグの修正 (Version 4.3 追記)

3. **テストプレイ開始時のキャッシュ残存によるチュートリアル未開始バグ修正**:
   - タイトル画面でのサインアウト時に `authToken.ts` 内のメモリ上のトークンキャッシュが残存し、再ログイン（テストプレイ開始）時に前ユーザーのプロフィール情報をフェッチしてしまう問題を解消するため、`TitlePageInner.tsx` のすべての `supabase.auth.signOut()` 実行直後に `clearAuthTokenCache()` を呼び出すように変更し、古いトークンキャッシュを強制破棄するように修正した。

## 19. 全モーダルコンポーネントの React Portal 化および iOS Safari タッチオフセットバグの徹底的解消仕様 (Version 4.4 追記)

1. **すべてのモーダル・オーバーレイの React Portal (`createPortal`) 化**:
   - iOS Safari (WebKit) の長年のバグである「スクロールするコンテナ（`overflow-y-auto`）内、またはスクロールする親と同階層に `fixed` 要素が存在する場合、スクロール量に応じてタップ座標（ヒットテスト）が上下左右にズレる問題」を根絶するため、すべてのモーダルを React Portal を利用して `document.body` 直下にレンダリングする設計に移行した。
   - Next.js の Hydration Mismatch (SSR/CSR の差異による表示エラー) を避けるため、各モーダルには `mounted` ライフサイクル管理を追加し、マウントが完了（`useEffect` 実行後）してから Portal レンダリングを行う。
   - **対象ファイル**:
     - `GuestRegisterPromoModal.tsx`
     - `StarterPackPromoModal.tsx`
     - `QuestResultModal.tsx`
     - `TavernModal.tsx`
     - `QuestBoardModal.tsx`
     - `NpcDialogModal.tsx`
     - `PrayerModal.tsx`
     - `AccountSettingsModal.tsx`
     - `StatusModal.tsx`
     - `BillingModal.tsx`
     - `ColosseumModal.tsx`
     - `ColosseumRankingModal.tsx`
     - `ActiveQuestModal.tsx`
     - `UgcQuestBoardPanel.tsx`
     - `HistoryArchiveModal.tsx`
     - `GossipModal.tsx`
     - `ConfirmDialog.tsx`
     - `page.tsx` (`VitalityDeathModal` インライン定義)

2. **`QuestResultModal` のスクロールレイアウト改修**:
   - `QuestResultModal` 全体をスクロールさせる（`overflow-y-auto`）構造が iOS WebKit のバグや座標ズレを助長していたため、最外枠のスクロールを廃止。
   - モーダルカードの最大高さを `max-h-[90dvh]` (ダイナミックビューポートハイト) に制限し、`flex flex-col overflow-hidden` を適用。
   - ヘッダーおよびフッターの「冒険を続ける」ボタン部を `shrink-0` で固定（Sticky）し、報酬アイテムやログを表示するコンテンツ部分のみを `flex-1 overflow-y-auto` とした独立スクロール構造へとリファクタリング。これによりスクロール時のタッチバグを完璧に回避しつつ、小画面デバイスでの操作性を劇的に向上させた。
