# spec_new_onboarding: 新規オンボーディングフローおよびおまかせデッキ編成仕様書

**Version**: 1.7  
**Date**: 2026-06-25  
**Status**: Planned  

---

## 1. 概要

新規ユーザーの初期離脱率低下およびデータ損失防止を目的とし、既存のオンボーディングおよびチュートリアル（宿屋での `TutorialModal`）を完全に廃止し、世界観説明 ➔ パック開封 ➔ おまかせデッキ編成 ➔ 第1話「始まりの轍」（シナリオID: `6001`）の強制開始までを一本道で繋ぐ「レール式オンボーディング」を導入する。  
また、第1話クリア後にはユーザーの属性（ゲスト / 本登録）に合わせたプロモーション表示を行い、その後のゲームサイクル（酒場雇用 ➔ ギルド受注 ➔ マップ出発）をステップバイステップでナビゲートする。

---

## 2. 開発の分離・競合防止方針

並行して実施されている可能性のある他の機能開発やクエスト改修とコードが競合するのを防ぐため、本開発は**完全な独立分離設計**で実装する。

1. **新規モジュール・エンドポイントの分離**:
   - チュートリアル用のパック開封画面は、既存の `AcademyModal.tsx` を一切改変せず、新規に作成する `OnboardingAcademyModal.tsx` で完結させる。
   - おまかせデッキ編成処理は、汎用のインベントリAPIを拡張するのではなく、専用エンドポイント `/api/inventory/auto-deck` を新設して処理する。
2. **コア処理エンジンへの介入排除**:
   - シナリオ処理やバトルなどの共通コアエンジン（`ScenarioEngine.tsx` や `BattleView.tsx`）のコードには一切手を加えず、画面遷移（Router）と宿屋画面（`/inn`）への帰還時のフラグ判定のみで本オンボーディングフローを完結させる。これにより、他のクエスト開発との競合影響を「ゼロ」とする。

---

## 3. 開発ルール (KI) への準拠

開発ガイドライン `AGENTS.md` の制約に基づき、以下の実装設計を適用する。

1. **アトミックなキー付与と重複防止 (ルール 12.3 準拠)**:
   - 新規登録時に「知識と契約の鍵 (ID: 76)」を付与する際、単なる `insert` ではなく `ON CONFLICT (user_id, item_id) DO UPDATE SET quantity = inventory.quantity + EXCLUDED.quantity` による upsert 制御を用いて安全に実行する。
2. **非同期通信時の二重発火・連打防止ガード (ルール 3.1 & 10.6 準拠)**:
   - `OnboardingAcademyModal.tsx` で「おまかせデッキ編成 ＆ 冒険を開始」ボタンがクリックされた際、非同期通信（自動編成 ➔ 受注 ➔ チュートリアル完了 ➔ 遷移）の処理中は、UI側でローカルステート `isProcessing` を `true` に設定し、ボタンを無効化（`disabled`）して二重送信を物理的にブロックする。
3. **限定パッケージの二重購入防止 (ルール 12.2 準拠)**:
   - `StarterPackPromoModal` を表示する際、フロントエンドで `userProfile.has_purchased_starter` および `userProfile.has_purchased_elite` を参照する。
   - 両方のパックを既に購入済みの場合は、モーダル自体の表示をスキップ（表示しない）する。
   - モーダル内部では、購入済みのパックを非活性化（「購入済み」と表示）する。
4. **決済パラメータキーの整合 (ルール 13.1 準拠)**:
   - 課金モーダルを起動する際は、Stripe Webhook 等で使用されている正確なパッケージキー `starter_pack` および `elite_pack` を渡して決済プロセスを実行する。

## 4. オンボーディング専用モーダル (`OnboardingAcademyModal`)

新規モーダル `src/components/inn/OnboardingAcademyModal.tsx` を実装する。  
本モーダルはチュートリアル未完了（`userProfile.is_tutorial_completed === false`）の際に自動表示され、完了するまで閉じる（Xボタンなど）ことはできない。

### 4.1 各フェーズの遷移

#### フェーズ 1: 世界観説明
- **UI内容**: ダークファンタジー調で装飾されたカードUI上に、世界観を伴う以下の説明テキストを表示する。
  > 「目覚めよ、名もなき旅人よ。  
  > 混沌と秩序が交錯するこの地を歩むには、運命を切り拓く力……すなわち『カード』の契約が必要となる。  
  > 学舎の門を叩き、古の契約を結ぶのだ。」
- **操作**: 「魔術学院へ進む」ボタンのみを表示。

#### フェーズ 2: チュートリアル用パック開封
- **UI内容**: 「黎明の知識と古の契約（`basic`）」パックの画像を表示。
- **制限**: ゴールドでの購入ボタンは完全に排除し、ユーザー作成時に付与される「知識と契約の鍵（`item_basic_key`）」でのみ引けるように制限。

#### フェーズ 3: 開封演出・カードめくり
- **UI内容**: 既存の `AcademyModal` と同様の引き裂きおよび5枚のカード表面めくり演出を表示。

#### フェーズ 4: デッキ自動編成・第1話開始
- **UI内容**: 開封完了後、「もう一度引く」「閉じる」ボタンを非表示にし、代わりに「おまかせデッキ編成 ＆ 冒険を開始」ボタンを配置。
- **制限**: 処理実行中はボタンを `disabled` に制御して連打によるAPI多重送信を防ぐ。
- **ボタン押下時の動作**:
  1. `POST /api/inventory/auto-deck` を呼び出し、所持カードから自動デッキ編成を実行。
  2. `POST /api/quest/start` を呼び出し、第1話「始まりの轍」（ID: `6001`）を受注。
  3. `POST /api/profile/complete-tutorial` を呼び出し、チュートリアルを完了。
  4. 完了後、自動的に `/quest/6001` へ画面遷移。

## 5. バックエンド API 仕様

### 5.1 `POST /api/inventory/auto-deck` (おまかせデッキ編成)
- **認証**: JWTトークンによるユーザー特定。
- **処理ロジック**:
  1. ユーザーの `max_deck_cost` を `user_profiles` から取得（デフォルト: 10）。
  2. `user_skills`（正規スキル）および `inventory`（レガシースキル）の所持カードをすべて取得。
  3. 一旦すべての装備フラグ（`is_equipped`）を `false` にリセット。
  4. カードを `deck_cost`（コスト）の降順にソート。
  5. 降順に走査し、`累計コスト + 当該カードコスト <= max_deck_cost` である場合、当該カードの `is_equipped` を `true` に設定（貪欲法）。
  6. 設定した装備状態をデータベースへ保存。
  7. 装備した正規スキルの `card_id`（最大6枚）を抽出し、`user_profiles.signature_deck` を更新。
  8. 成功時に `{ success: true, current_deck_cost }` を返却。

### 5.2 `POST /api/profile/init` (初期鍵の付与)
- **処理ロジック**:
  - 新規プロフィール登録（データベースへのINSERT）が行われる際、自動的に `inventory` テーブルに「知識と契約の鍵（`item_id`: 76, 数量: 1, `is_equipped`: false）」をインサートする。
  - SQL レベルで衝突を避けるため、**ON CONFLICT (user_id, item_id) DO UPDATE SET quantity = inventory.quantity + EXCLUDED.quantity** の upsert ロジックを使用する。

## 6. クリア後の属性別プロモーション

第1話クリア後に宿屋に戻った際、ユーザー種別に応じた案内を行う。

### 6.1 ゲストユーザー向け：本登録案内
- **トリガー**: 第1話（または他クエスト）クリア後、ユーザーがゲスト（`is_anonymous === true`）である場合。
- **表示内容**:
  - 新規生成アセット `promo_guest_register.png` をヘッダーに配置。
  - 世界観に沿った以下の案内メッセージを表示：
    > 「旅人よ、素晴らしい活躍であった。だが、忘れてはならない。  
    > 署名なき冒険譚は風に消えゆく宿命にある。  
    > お前の歩んだ確かな軌跡を、この世界の歴史書に深く刻み込むのだ。  
    > さすれば、お前の冒険の記録は永遠に失われることはないだろう……」
  - 「Googleアカウントで冒険の記録を残す（本登録）」ボタンをタップすると、既存の Google アカウント連携（`linkIdentity`）処理を実行。

### 6.2 本登録完了直後の連携遷移（ゲスト ➔ 本登録）
- **動作フロー**:
  1. ゲストユーザーが本登録（OAuth連携）を行う前に、`sessionStorage.setItem('wirth_dawn_just_registered', 'true')` を設定する。
  2. Google等の外部連携手続きを経て宿屋（`/inn`）へリダイレクトバックされる。
  3. `sessionStorage` に `'wirth_dawn_just_registered'` が存在し、かつ `userProfile.is_anonymous` が `false`（本登録ユーザー）に切り替わっていることを検知。
  4. 検知後、セッションフラグをクリアした上で、**自動的にスターター/エリートパック案内モーダル (`StarterPackPromoModal`) をポップアップ起動する**。
- **目的**: ゲスト登録を完了したユーザーを離脱させず、そのままプレミアムパックのプロモーションにシームレスに繋ぐ。

### 6.3 本登録ユーザー向け：スターター/エリートパック案内モーダル (`StarterPackPromoModal`)
- **トリガー**: 
  - ① 第1話クリア後の帰還時（すでに本登録済みの場合）。
  - ② ゲストユーザーが本登録を完了して宿屋にリダイレクトバックされた直後（上記 6.2 のフロー）。
  - ③ 本登録済みかつ第1話クリア済みのユーザーが「次回ログイン」した時（`sessionStorage` でセッション中1回のみポップアップするよう制御）。
- **表示内容**:
  - 新規生成アセット `promo_starter_pack.png` を配置。
  - **購入状態のチェック**: `userProfile.has_purchased_starter` / `has_purchased_elite` を参照し、購入済みのものは「購入済み」と非活性表示する（双方購入済みの場合はモーダル表示自体をスキップ）。
  - **世界観に沿った案内メッセージ**:
    > 「見事であった。  
    > 険しさを増すこれからの旅路に備え、旅人の力となる『スターターパック』、そして深淵なる知識を秘めた『エリートパック』に興味はあるか？興味があるのであれば、内容を確認し、汝のデッキをさらに強固なものとするが良い……」
  - **ボタンの選択肢と遷移**:
    * **「内容を確認する」ボタン**: タップ時にモーダルを閉じ、既存の課金画面（`BillingModal`）を `packageKey` ('starter_pack' または 'elite_pack') を添えて起動する。
    * **「すぐに冒険を開始する」ボタン**: タップ時にモーダルを閉じ、次のステップ（酒場での仲間雇用ナビ）に進む。

## 7. 冒険の旅立ちナビゲーション (目的ガイダンスバナー)

第1話クリア後、プレイヤーが段階的に拠点の全貌と主要なシステムを理解できるよう、ガイダンスバナーとハイライトによる動的なステップナビゲーションを7段階で提供する。
宿屋ページに各訪問状態を管理するフラグ（`localStorage` に保存）を追加し、以下の優先度で状態を遷移する。

1. **① 宿屋/酒場での仲間雇用ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 0人
   - バナー: 「💡 旅の目的: 「宿屋/酒場」で最初の仲間（NPC）を雇用しましょう！」
   - 推奨施設: `inn` (宿屋/酒場) をハイライト。
2. **② ギルドでのクエスト受注ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 1人以上 ＆ ギルド未訪問 (`!visitedGuild`)
   - バナー: 「💡 旅の目的: 「ギルド」で新たな依頼（クエスト）を引き受けましょう！」
   - 推奨施設: `guild` (ギルド) をハイライト。
3. **③ ワールドマップでの冒険開始ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 1人以上 ＆ ギルド訪問済 ＆ マップ未訪問 (`!visitedMap`)
   - バナー: 「💡 旅の目的: 「出発する」を押して、ワールドマップから冒険に旅立ちましょう！」
   - 推奨ボタン: メインビジュアルエリアの `ワールドマップを開く` (Compass) ボタンをハイライト。
4. **④ 魔術学院でのカード契約確認ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 1人以上 ＆ ギルド訪問済 ＆ マップ訪問済 ＆ 魔術学院未訪問 (`!visitedAcademy`)
   - バナー: 「💡 旅の目的: 「魔術学院」で新たな魔導の契約（カードパック購入）やデッキの確認をしましょう！」
   - 推奨施設: `academy` (魔術学院) をハイライト。
5. **⑤ 道具屋・チャージの確認ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 1人以上 ＆ ギルド訪問済 ＆ マップ訪問済 ＆ 魔術学院訪問済 ＆ 道具屋/チャージ未訪問 (`!visitedShop` または `!visitedBilling`)
   - バナー: 「💡 旅の目的: 「道具屋」や「チャージ」を開いて、冒険に役立つアイテムや金貨を確認しましょう！」
   - 推奨ボタン/施設: `shop` (道具屋) の施設ボタンおよびヘッダーのゴールドチャージボタンをハイライト。
6. **⑥ ステータス画面の確認ナビ**:
   - 条件: 第1話クリア ＆ 同行メンバー 1人以上 ＆ ギルド訪問済 ＆ マップ訪問済 ＆ 魔術学院訪問済 ＆ 道具屋/チャージ訪問済 ＆ ステータス未確認 (`!visitedStatus`)
   - バナー: 「💡 旅の目的: 「ステータス」を開いて、キャラクターの能力値や装備品を確認しましょう！」
   - 推奨ボタン: ヘッダーまたはメニューの `ステータス (Status)` 関連ボタンをハイライト。
7. **⑦ 各種設定の確認ナビ**:
   - 条件: 上記①〜⑥すべて完了 ＆ 設定未確認 (`!visitedSettings`)
   - バナー: 「💡 旅の目的: 「設定（歯車マーク）」を開いて、音量調節やアカウント管理の機能を確認しましょう！」
   - 推奨ボタン: ヘッダーの `設定 (Cog)` ボタンをハイライト。

## 8. 号外システムの一時廃止

ユーザー体験向上のため、以下の号外新聞シェア機能および画像生成処理を停止する。
- **UI側**: `QuestResultModal.tsx` 下部の「号外！冒険を共有しよう」セクション（SNSシェア機能）をレンダリングから除外。
- **API側**: `/api/quest/complete` および `/api/character/retire`, `/api/profile/init` 内の `share_data_list` の生成処理を抑止し、常に空の配列（`[]`）を返す。
