**Version**: 1.0

**Status**: Implemented & Verified

## 1. 概要

ハブ拠点（名もなき旅人の拠所）を除くすべての通常拠点において、新施設「魔術学院（Magic Academy）」を実装する。
プレイヤーはゴールドを消費して「混沌の魔導と反逆の鉄壁」ブースターパックを購入し、新たに導入された40種のスキルカードを獲得することができる。

### 1.1 基本ルール

- **購入費用**: 1パック 5,000G
- **パック内容物**: スキルカード5枚
- **重複救済（キャッシュバック）**: パック内で重複したスキル、またはすでに習得済みのスキルを獲得した際、1枚あたり **500G** が購入費用から相殺・返還される（キャッシュバック処理）。
- **名声による制限（出禁）**: 滞在している拠点における名声値が **`-300` 以下** の場合、施設への立ち入りおよびパック購入が拒否される（出入り禁止状態）。

## 2. データベースおよびガチャ仕様

### 2.1 除外対象カード
以下の3枚のカードは「闇市」限定の強力な禁忌カードであるため、魔術学院のガチャ（ブースターパック）の排出プールからは厳密に除外される。
- **`3121` (死神の宣告 / Death Sentence)**
- **`3129` (成金の一撃 / Pay to Win)**
- **`3132` (破滅の契約 / Ruin Pact)**

### 2.2 ガチャ確率と排出プール

ブースターパックの排出確率はスロット（枚数）に応じて以下のように制御される。

| スロット | SR（スーパーレア） | R（レア） | U（アンコモン） | C（コモン） |
|:---|:---|:---|:---|:---|
| **1〜4枚目** | 3% | 12% | 25% | 60% |
| **5枚目 (R以上確定)** | 20% | 80% | 0% | 0% |

## 3. バックエンド API 仕様

### 3.1 `POST /api/shop/buy-pack`

パックの購入、抽選、重複チェック、およびゴールドの減算をアトミックに実行する。

- **セキュリティ**: JWT認証トークン必須。`getAuthenticatedProfile` を使用し、トークンに紐づく `user.id` からプロフィールを取得。
- **ハブ拠点チェック**: プレイヤーの現在位置が「名もなき旅人の拠所（Hub）」である場合は購入エラー（400 Bad Request）を返す。
- **出禁判定**: `checkEmbargo` を通じて該当拠点の名声値が `-300` 以下の場合は 403 Forbidden を返す。
- **ゴールド減算の排他制御**: 重複によるキャッシュバック分（1枚につき500G）を算出した後、アトミックな `increment_gold` RPCを実行して二重消費や負数残高を防止。
- **スキル登録**: 新規に獲得したスキルは `user_skills` テーブルへ登録される。

## 4. フロントエンド UI & 演出仕様

### 4.1 拠点への配置
- **表示ロジック**: `FacilityGrid.tsx` 内で、ハブ（`isHub = true`）以外の通常拠点（`isHub = false`）の施設メニューに「魔術学院」ボタンを追加。
- **NPCダイアログ**: 担当NPC「テオドール」が常駐し、名声レベルに応じたセリフ（高名声・通常・低名声・出入り禁止）を表示。出禁時はメインアクションボタンが「出入り禁止」表示となり非活性化される。

### 4.2 購入・開封演出 (`AcademyModal.tsx`)
- プレミアム感のある開封体験を提供するため、Vanilla CSS と React を用いたアニメーションを構築。
- **開封前**: 未開封のブースターパックを3D表示。パックをタップすることで「撕裂（引き裂き）」アニメーションが走り、光のピン演出とともにカード出現フェーズへ遷移。
- **カードめくり**: 5枚のカードが裏向きで並ぶ。各カードをタップするとフリップ動作（3D回転）が行われ、レア度に応じたSEおよび発光エフェクト（SR: 紫、R: 金、U: 青）が再生される。
- **重複表示**: 重複カードには「ダブり」のバッジと「+500Gキャッシュバック」のオーバーレイが表示される。

## 5. 特殊スキルカードの戦闘ロジック

追加されたスキルカードのうち、以下の5枚には特殊な処理エンジンを `battleSlice.ts` 内に個別に組み込んでいる。

1. **カタルシス (ID: 101)**
   - 対象が「毒（poison）」または「炎上（burn）」状態である場合、残りの全ターン分のスリップダメージ（毎ターン最大HPの5%）を即座に起爆ダメージとして与え、その後対象の「毒」「炎上」デバフを解除する。
2. **シールドスラム (ID: 105)**
   - プレイヤーが受けている防御バフ（`def_up`, `def_up_heavy`, `barrier`）の数値を加算したダメージ（`10 + 防御バフ値`）を対象に与え、発動後に自身の防御バフを全て解除（消費）する。
3. **成金の一撃 (ID: 129)**
   - 使用時に所持ゴールドの 2% （最大 1,000G）を消費する。消費したゴールド / 10 の追加ダメージを上乗せして与え、ゴールド消費は即座に DB 側（`/api/profile/update-status`）に同期される。
4. **ダブルキャスト (ID: 119)**
   - 使用時に自身に「二重詠唱（double_cast）」状態を付与。次に使用する Magic または Skill タイプのカードは、消費AP 0 で2回連続発動（ダメージ・バフ・自傷・ゴールド消費等の効果が順次ループ実行）する。
5. **サーチライト (ID: 118)**
   - デッキからランダムな「Heal（回復）」または「Defense（防御）」タイプのカードを1枚抽出して手札に加え、デッキ（山札）から削除する。

## 6. ボスの即死耐性の導入

即死攻撃がボス戦闘において強力すぎる問題を解決するため、即死耐性（`death_immune`）の概念を導入する。

### 6.1 即死耐性フラグ
- `enemies` テーブルに `death_immune`（boolean）カラムを追加。
- ボス・エリート・ハンター・神などの強力なエネミー（全78種）には `death_immune = true` を設定。通常の雑魚エネミーには `false` または `null` を設定。

### 6.2 即死スキルの耐性判定
1. **即死攻撃 (Card ID: 58 / `instakill`)**
   - 対象が `death_immune = true` の場合、即死判定をスキップし、代わりに通常の物理ダメージを与える。
   - 戦闘ログ: `[敵名]に即死攻撃を使用！ 即死を無効化されたが、[ダメージ値] のダメージ！`
2. **死神の宣告 (Card ID: 121 / `death_sentence` デバフの5T終了時)**
   - 対象が `death_immune = true` の場合、即死処理をキャンセルし、代わりに **最大HPの20%（最低300ダメージ）の割合ダメージ** を与える。
   - 戦闘ログ: `💀 [敵名]は即死耐性により即死を無効化し、代わりに [ダメージ値] の大ダメージを受けた！`

## 7. パフォーマンス最適化（ログ表示のコンポーネント分離）

バトルの演出追加やカード効果拡張に伴うレンダリング負荷を軽減するため、UI構造の最適化を実施する。

### 7.1 タイプライター処理のコンポーネント分離とメモ化
- **問題点**: ログの1文字ずつのタイプライター表示処理（20msタイマー）の毎回のステート更新により、約115KBある `BattleView.tsx` 全体が毎秒50回再レンダリングされ、動作のもたつきを引き起こしていた。
- **対策**:
  - `typingText` 状態およびタイプライター表示処理を `BattleView.tsx` から完全に剥離。
  - 新規子コンポーネント **`BattleLogBox.tsx`** （`React.memo` でラップ）を導入し、タイプライタータイマー処理（`TypewriterLine`）をその内部にカプセル化。
- **効果**: 文字描画時の再描画スコープがログ表示部分のみに限定され、手札の再計算や背景描画の負荷がゼロになり、戦闘全体のFPSと操作応答性が大幅に向上。
