# Wirth-Dawn 開発・運用ルール (KI)

## 1. メンテナンスモードおよび Edge Middleware に関する技術的制約
- **Edge Middleware における JWT デコードの注意点**:
  - Edge Runtime (Next.js Middleware 等) では、Base64Url のデコードを行う `atob` がパディング `=` を欠いている場合に `DOMException` をスローするため、必ず事前に `while (base64.length % 4) { base64 += '='; }` などのパディング補正処理を挟むこと。
  - Supabase SSR が書き込むクッキー（`sb-*-auth-token`）はサイズが大きい場合に自動で分割保存（`*.0`, `*.1` 等）され、かつ値全体が Base64 エンコードされている場合がある。そのため、`auth-token` を名前に含むすべてのクッキーを収集・ソート結合した上で、JSON パースの前に一度 Base64 デコード処理を行うロジックを維持すること。
- **サーバーコンポーネント内でのイベントハンドラ厳禁**:
  - Next.js のサーバーコンポーネント（特に `/maintenance` などの独立画面）では、HTML 要素に `onMouseOver` や `onMouseOut` などのイベントハンドラを直接渡してはならない。これを行うと、サーバーレンダリング時にクラッシュ（500エラー）を引き起こす。
  - 代替として、`<style>` タグや Pure CSS (`:hover` クラス) を使用してスタイリングのインタラクションを実現すること。
- **タイムゾーンの明示的指定**:
  - メンテナンス画面などの日付表示では、サーバーが UTC タイムゾーンで動作している場合があるため、`new Date()` の `getHours()` を直接呼ぶとずれた時刻が表示される。必ず日本時間 (JST: UTC+9) のオフセットを明示的に加算し、`getUTC*` 系のメソッドを用いてフォーマットすること。

## 2. リリース検証およびメンテナンス手順
- **事前アカウント検証 of 徹底**:
  - メンテナンス（および緊急リリース）のデプロイ時には、必ず事前に登録されているホワイトリストアカウント（きたむ様アカウント。新ID: `5ad434ec-763f-473e-939f-14a5e9e1cc93`、旧ID: `c1cf67dd-527a-497e-bf88-ce10c2cb516f`）での動作確認を徹底すること。
  - **次回以降のメンテナンス検証時、必ず「きたむ様アカウント」で実際にログインし、正常にメンテナンスをバイパスしてゲームに入れることを本番環境で徹底検証する。**
  - **Googleなどの OAuth 認証が行われる前段階（認証クッキーが存在しない状態）**では自動バイパスが機能しない。そのため、以下の2ステップで検証を行うこと：
    1. **未認証状態**: 管理者用バイパス URL（`?bypass=<bypass-key>`）を経由させてブラウザにバイパス用クッキーを植え付け、メンテナンス画面を通過してログイン画面が表示されるか確認する。
    2. **認証完了状態**: 実際に「きたむ様アカウント」でGoogleログインを行い、メンテナンス制限に引っかからずに拠点（宿屋等）に入れるか、一般ユーザーが遮断されている状態で正常にゲームをプレイできるか確認する。
  - **コロシアムランキングにおける除外仕様**: きたむ様アカウント（新旧ID）は、コロシアムランキングのAPIレスポンスから自動的にフィルタリング（除外）されるよう実装されている。そのため、ランキング画面での表示テストや集計動作検証を行う場合は、別途一般テストアカウントを使用する必要がある。
- **環境間の逆マージ (バックポート) の確認**:
  - 本番環境（`main`）に直接適用されたマイグレーションやソースコードがある場合は、放置せず速やかに開発環境（`develop`）および PREVIEW DB へ逆マージし、環境間の不整合を防ぐこと。

## 3. 非同期処理と状態クリアに伴う競合（連打バグ）の防止
- **フロントエンドにおける多重遷移ガードの徹底**:
  - バトルの終了（勝利・敗北・撤退）時やクエスト結果の確定時など、サーバー通信を伴う非同期処理が行われるボタンは、必ずフロントエンド側で多重実行防止フラグ（`isTransitioning` / `isProcessingEndRef` 等）を導入し、ボタンの `disabled` 化と二重発火防止ガードを行うこと。
  - 非同期処理の中でメモリ上の遷移先情報（`successNodeIdRef.current` 等）をクリアする場合、クリアタイミングと再実行の競合により、クリア後の `null` が読み取られて安全性の低いフォールバック先（無条件でのクエストクリアノード等）へワープする恐れがある。
- **安全なフォールバック設計**:
  - 状態が不正クリアされた場合のフォールバック先は、無条件の成功・クリアノード（`end_success` 等）に設定してはならない。必ず直前の正常状態（バトル開始前のノード、あるいは戦闘発生マスなど）に戻すようフォールバック設計を徹底すること。
- **完了APIにおけるバトル検証トークンの必須化**:
  - コロシアムなどの連戦・特殊クエストであっても、完了API（`/api/quest/complete`）でバトル検証トークンによる署名チェックがスキップされないよう、モック定義時にも適切な `script_data.nodes` を構築し、サーバーサイドでの改ざん・ワープ防止策を確実に作動させること。

- **クエスト状態のローカルキャッシュ初期化**:
  - クエストページ（`page.tsx`）初期化時、サーバー側の `current_quest_state` が `null` である場合は、ブラウザのZustand永続化キャッシュ（`quest-storage`）の状態に関わらず、必ず `startQuest` を呼び出してローカルの進行データを強制リセットし、ゾンビデータによる進行度のズレや同期上書きを防ぐこと。

- **Zustand セレクタによるパフォーマンス最適化**:
  - 巨大なコンポーネント（特に `BattleView.tsx` などの大規模UI）では、Zustandの `useGameStore()` をセレクタなしで呼び出してはならない。ストア全体の不要な更新を監視してしまい、超高頻度の再レンダリングが発生して動作がカクつく原因となるため、必ず個別にセレクタ形式（`state => state.property`）で必要なプロパティだけを個別に切り出して参照すること。

- **バトル撃破とアニメーション演出の同期**:
  - 敵の死亡表示（スプライトのモノクロ・ぼかし化）は、攻撃エフェクトのアニメーション（`activeEffect`）が再生されている途中で即時反映されてはならない。エフェクトの再生終了（`activeEffect` のクリア）を待って生存状態を維持させ、演出のぶつ切りを防止すること。

- **バトル終了時のログ再生完了ガード**:
  - 敵を全滅させてバトルが勝利した際、Zustand store で `isVictory` が即座に `true` に更新されるが、フロントエンドのメッセージ同期とタイプライター再生が完了する前に Victory 画面がオーバーレイ表示されてしまってはならない。
  - タイマーによるポーリング判定は非同期な状態更新（`validate-result` や `userProfile` の更新など）と競合して誤検知しやすいため、実際にタイプライター表示完了したログ配列数 (`displayedLogs.length`) と、ストアから届いた全メッセージから非表示の同期用メッセージを除外した実テキストログ数 (`battleState.messages.filter(m => !m.startsWith('__')).length`) を直接比較して、双方が一致するまで遷移をガードする実装規約を徹底すること。

## 4. ステータス・スキル・装備品モーダルの設計原則 (v28)
- **タブUIの廃止と2x2グリッド配置**: ステータスモーダルにおけるタブ切り替えUIは廃止され、生命力バーの下に2x2グリッド（スキルデッキ、装備品、所持品、パーティ）のボタンを配置し、それぞれ独立したサブモーダルをオーバーレイ表示（createPortal）する。
- **アコーディオン（折りたたみ）とフィルタ**:
  - 装備中のスキルおよび装備中のスロット（武器、防具、アクセサリー）は折りたたまず、常時展開して表示すること。
  - ストック一覧（所持スキル、NPC支援カード、武器、防具、アクセサリー）はカテゴリごとに折りたたみ（開閉）可能とすること。
  - 所持スキルストックには、コスト（All, C0, C1, C2, C3+）によるフィルタコントロールを提供すること。
- **クエスト進行中の変更制限**: クエスト進行中は、スキルや装備品、およびパーティメンバーの並び替え（ソート順）の変更を禁止する制御（`questLocked` などによるロック）を維持・徹底すること。
- **パーティメンバーの並び替え機能 (v28.1)**:
  - パーティモーダルでは、同行NPC of 並び順を変更する「▲」「▼」ボタンによるソート機能を提供し、データベースの `party_members.sort_order` と同期させること。
  - バトル時のNPC支援カードの注入優先順や、身代わり防御順（左＝先頭メンバー優先）を意図通りコントロールできるよう正確な順序管理を行うこと。

## 5. 進行不能不具合に対する補填およびデータ抽出手順
- **被災ロケーションの特定**:
  - クエストやコロシアムの遷移ロック等の不具合により、ユーザーがリタイアせざるを得なかった場合、その被災時のロケーション名（`location_name`）やペナルティ値は、`user_chronicles` テーブルの `event_type = 'quest_failure'` またはそれに類する失敗・放棄タイトルに紐付くレコードから特定すること。
  - `user_chronicles` からの抽出クエリは、Supabaseのデフォルトの 1000 件取得制限を回避するため、必ずユーザーID単位で個別にクエリをループ実行し、漏れなくデータを取得すること。
- **補填データの安全な適用と自動アサーション**:
  - 補填実行スクリプト（`apply_compensation.js` 等）では、適用前の値（事前値）と補填値、および適用直後に再クエリした事後値を比較し、「事前値 + 補填値 === 事後値」がゴールド、名声スコア、インベントリ数量等の全項目で成り立つことをプログラム的に自動アサーション検証する規約を徹底すること。

## 6. クエスト内一時アイテムの所持判定 (check_delivery / check_possession) に関する実装規約
- **判定対象の拡張**:
  - クエスト進行中に `reward` ノード等で獲得した一時アイテムは、クエスト完了（`/api/quest/complete`）までデータベース（`inventory` テーブル）に永続化されず、フロントエンドの戦利品プール（`questState.lootPool`）に一時蓄積される。
  - そのため、クエストの道中で `check_delivery`（納品判定）や `check_possession`（所持判定）を行うノードでは、必ず **「データベースのインベントリ所持数」と「戦利品プール内の同名アイテム数」の両方を合算して判定する** こと。これを怠ると、一時アイテムの所持判定が必ず失敗する不具合が発生する。

## 7. 同行NPCの耐久度・HP・VITの表示および摩耗に関する設計原則 (v28.3)
- **耐久度(durability)とバトルHP、VITの区別とマッピング**:
  - `party_members` テーブル of `durability` はNPC of 寿命であるVIT（Vitality）であり、最大値はすべてのNPCで一律 **`100`** 固定（初期値100）である。
  - クエスト完了ごとに、成否や戦闘不能 of 有無に関わらず、各パーティメンバー個別に **`1〜5` のランダム値** が減算され、0以下になったNPCは離脱（物理/論理削除）する。
  - `max_durability` は雇用時の最大HPスナップショットであり、戦闘時のHP上限である。HP of 現在値データはデータベースに保存せず、非戦闘時（街やステータス画面）は常に最大HPで全快表示される。
  - クエスト進行中の連続戦闘時のみ、戦闘終了時の残りHPがクライアント側のZustandストア（`questState.partyHp`）に一時退避され、連戦時の初期HPとしてマージされる。また、クエスト中の非戦闘UI（移動画面やステータス画面）でもこのZustandの値がマージされて現在HPとして表示される。
  - 宿屋の宿泊（`/api/inn/rest`）の際は、戦闘不能メンバーの生存フラグ（`is_active = true`）を復帰させるのみとし、寿命（`durability`）は全回復させない。
  - クエスト結果画面での寿命減少表示は、「耐久 100 ▸ 97」のように「耐久」ラベルを使用して摩耗を通知する。
- **離脱時の形見（メメント）の生成とフォールバック**:
  - 寿命（durability）が0以下になったNPCが離脱する際、その第1固有スキル（`inject_cards[0]`）に対応する `items` テーブルのアイテム（`linked_card_id` で紐づくもの）を形見アイテムとしてインベントリに付与する。
  - 対応するアイテムが存在しない基本スキル（石投げ、防御、強打など）の場合、フォールバックとして「物資ボックス (ID: 701, slug: 'item_supply_box')」を付与する。

## 8. バトル勝利時の非同期処理によるUI制御（ブロッキング回避） (v29.0)
- **非同期・非ブロッキング処理の原則**:
  - バトル終了時の勝利処理（`/api/report-action`, `/api/battle/validate-result`, `/api/profile/update-status` の呼び出しや、`fetchWorldState`, `fetchUserProfile` による再取得など）は、UI表示（メッセージや勝利画面）を妨げてはならない。
  - バトル進行アクション（`runNpcPhase` 等）の中でこれらの非同期通信を `await` してしまうと、ネットワーク環境やDB書き込みの遅延がそのままテキストログ表示のタイムラグや画面硬直となって现れる。
  - そのため、勝利確定時は即座に `isVictory: true` やログメッセージをステートに `set` して画面描画を更新し、上記のデータベース更新やサーバー検証APIは `await` せずにバックグラウンド（Promise または `.then()` チェーン）で並行実行する設計を徹底すること。

## 9. 閲覧施設（コレクション・クエスト記録・ランキング）の配置設計原則 (v30.0)
- **共通の表示と閲覧**:
  - 「コレクション（図鑑）」「クエスト記録」「ランキング」の3つの施設は、旅人の拠所（ハブ）だけでなく、通常拠点（宿屋/酒場）のメニューにも一貫して表示させる。
  - これらは閲覧専用 of 施設であり、いずれの拠点から開いた場合でも動作やデータ取得のロジックに差分を設けてはならない。共通の API およびモーダルコンポーネントを流用して表示すること。
- **世界情勢変化ポップアップの非表示・廃止**:
  - 6時間ごとの世界情勢変化時に拠点アクセスした際、自動で表示されていた警告ポップアップ（`WorldChangedModal`）および「号外（`ChronicleModal`）」の自動ポップアップ表示は廃止されたため、今後も復活させてはならない。過去の情勢変化の履歴は、歴史館（`HistoryArchiveModal`）からのみ閲覧可能とする。

## 10. 気の癒やし・デトネーション・HP装備補正維持に関する仕様規約 (v35)
- **気の癒やし（回復+リジェネの同時適用）**:
  - `case 'heal'` アクション内で `effectInfo.effectId`（リジェネ等）が存在する場合、回復と同時にバフの付与も行うこと（`skipDamage: true` 時の共通バフ付与スキップ仕様と干渉させない）。
  - NPCのターン処理（`processPartyTurn`）では、アクションが `buff` であっても、紐づくカードが回復タイプの場合は回復処理（`healAmount`）も並行して適用すること。
  - NPCが自分以外の特定メンバーに単体バフ・回復をかける際、そのバフが caster (発動者自身) ではなく、正しく `action.targetName` の味方NPCに適用されるようターゲット名と順序を特定して適用すること。
- **デトネーションのミス時手札破棄**:
  - 攻撃ミス判定（`rollMiss`）が成功して早期リターン（`return true`）する場合であっても、カードの `effectType` が `detonation` の場合は手札からのランダム2枚除外（Exhaust）を漏れなく発動させ、除外されたログをメッセージに追加すること。
- **クエスト進行を通じたHP装備補正値の維持**:
  - クエスト開始パラメータ（`startParams`）の `playerMaxHp` は、ベース最大HPではなく、必ず装備HP補正値（`equipBonus.hp`）を加算した実効最大HPとして初期化すること。
  - クエスト完了時（`/api/quest/complete`）のレベルアップ更新（`updates.hp`）およびクエスト失敗ペナルティによるHP復旧時、およびクエスト内の `healParty` による回復上限キャップ時に、装備HP補正分を切り捨てずに、新最大HP＋装備補正値の範囲で現在HP（`hp`）を復帰・維持すること。
- **スキルデッキの装備・装備解除（外す）における連打・重複処理の防止**:
  - スキルデッキ管理（`SkillDeckModal`）において、スキルの装備・装備解除（外す）アクションを実行する際は、非同期リクエストの実行中（`togglingId !== null`）すべてのアクションボタンを `disabled` にすること。
  - 重複リクエストや連打による多重送信を防ぐため、実行処理の先頭で `togglingId` の有無による早期リターン等の排他制御（連打防止）を行うこと。

## 11. 魔術学院機能拡張（黎明の知識と古の契約・鍵アイテム・スライドUI）に関する仕様規約 (v37)
- **既存スキルへのレアリティ定義**:
  - ベーシックパック収録カードだけでなく、すべての既存スキル（ID 3011〜3088, 3141〜3142）に対して、共通定義ファイル `src/lib/rarity.ts` 内で SR/R/U/C のレアリティマッピングを定義する。DBのスキーマ変更は行わない。
- **鍵アイテムのショップ除外と名称・説明の定義**:
  - パック開封用の鍵アイテムである `item_basic_key` (ID 76) および `item_academy_key` (ID 77) は、通常ショップに並ばないよう `items.csv` 上の `nation_tags` を空とし、さらにショップAPI `SHOP_EXCLUDE_SLUGS` に登録して除外を徹底する。
  - `item_basic_key` の名称は「知識と契約の鍵」、説明文は「カードパック「黎明の知識と古の契約」を開封することのできる鍵」とする。
  - `item_academy_key` の名称は「魔道と知識の鍵」、説明文は「カードパック「混沌の魔道と反逆の鉄壁」を開封することのできる鍵」とする。
- **ベーシックパック「黎明の知識と古の契約」の基本仕様**:
  - 販売価格は `3,000 G`、重複カード出現時のキャッシュバック返還額は `300 G` とする。
  - ベーシックパックのフリップ演出におけるカード裏面画像は、専用の青銅魔法円デザイン（`/images/card_back_basic.png`）を適用して他シリーズと分離する。
- **魔術学院UIの切り替えおよびカードリスト仕様**:
  - パックシリーズの切り替えは、今後の拡張性を見据えて従来のタブ切り替えではなく、左右の矢印ボタン（またはスワイプ）によるカルーセル（横スライド）方式とする。
  - 各パックのグラフィックの右下に「カードリスト確認」のアイコンボタンを配置し、クリックで収録カードがレアリティ（SR/R/U/C）ごとに整理されたグリッドギャラリーをモーダル上に表示させること。

## 12. マネタイズおよび課金統合モーダルに関する開発ルール (v38)
- **課金UIの統合とカプセル化**:
  - 課金、プラン加入、都度購入に関するすべてのUIは `BillingModal.tsx` に集約・カプセル化すること。設定画面などの他のモーダルに直接課金入力や購入ボタンを分散配置してはならない。
  - 特商法に基づく表示への同意チェックボックス（利用規約、返金不可などの確認）を必ず決済開始前にチェック必須にすること。
- **アカウント限定パッケージの二重購入防止**:
  - スターターパック・エリートパック等の「アカウント1回限り購入可能」な商品は、以下の3レイヤーすべてで厳密にガードをかけること：
    1. **フロントエンド**: `userProfile.has_purchased_starter` / `has_purchased_elite` フラグを参照し、購入済みの場合はボタンを非活性化（「購入済み」と表示）。
    2. **Checkout API (`/api/billing/checkout`)**: JWT認証でユーザーを特定後、DBから対応する購入フラグを直接取得してチェックし、すでに `true` の場合は `400 Bad Request` を返す。
    3. **Stripe Webhook (`/api/webhooks/stripe`)**: 決済完了処理の中で、PostgreSQL RPC (`process_package_purchase`) を呼び出す。このRPC内部で、購入フラグがすでに `true` の場合はトランザクションを即座にロールバック（または早期リターン）するアトミックなガードを行う。
- **ゴールドと鍵のアトミックな付与処理**:
  - サブスクリプションのWeeklyボーナス（ゴールド＋鍵）や限定パッケージ購入時のアイテム付与は、DBレベルのトランザクションを保証するため、フロント側やAPI側で個別に複数回クエリを実行してはならない。必ず PostgreSQL RPC (`process_weekly_subscription_bonus` / `process_package_purchase`) を介してアトミックに実行すること。
  - 鍵アイテム（`item_basic_key` (ID: 76) や `item_academy_key` (ID: 77)）をインベントリに付与する際は、必ず `inventory` テーブルに `quantity` 加算を行う upsert ロジック（`ON CONFLICT (user_id, item_id) DO UPDATE SET quantity = inventory.quantity + EXCLUDED.quantity`）を使用し、レコードの重複や上書き消失を防ぐこと。
- **魔術学院等のモーダル連携**:
  - 魔術学院 (`AcademyModal.tsx`) など、他のモーダルや画面から課金画面を起動する場合は、宿屋の共通ページステート (`useInnPageState.ts` や `page.tsx`) を経由して `BillingModal` を上に重ねてレンダリングする（オーバーレイ表示）設計とすること。モーダルを閉じたあとに元のモーダル（魔術学院等）に戻れるよう、モーダルのZ-indexやオーバーレイ順序を適切に制御すること。
- **既存加入者のアップデート時一括補填**:
  - 特典アップデート等により、既存のサブスクリプション加入者に対してアイテムの一括補填が必要な場合は、`supabase/migrations/` 配下にデータ移行用の移行SQLファイルを作成し、デプロイ時に一度だけ安全に実行する設計とすること。

## 13. 決済完了ダイアログのパラメータ判定ミスマッチ防止およびテスト用データのクリーンアップルール (v40)
- **決済完了パラメータのマッチング一貫性**:
  - Stripe 決済成功後に宿屋へ戻るリダイレクトクエリパラメータ（`starter_pack` / `elite_pack` など）と、フロントエンド（`useInnPageState.ts`）でマッピング判定されるパッケージ名（旧名: `gold_starter` / `gold_elite` など）がズレていると、購入した鍵などの特典がポップアップに表示されなくなる。
  - フロント側では、Stripe Webhook で使用されるキーと古い判定キーの双方を許容（`packageKey === 'starter_pack' || packageKey === 'gold_starter'`）するなどの後方互換性とミスマッチ防止策を二重に挟んで検知ロジックを構成すること。
- **アカウントデータ減算時の安全境界ガード**:
  - テストアカウントのリセットや、スターター/エリートパック購入取り消し時のゴールド・アイテムの減算を行うスクリプト/機能においては、計算によって所持金やアイテム数量がマイナスにならないよう、下限値（0以上）を担保すること（例: `Math.max(0, currentGold - subtractAmount)`）。
- **テスト用サブスクリプションの完全リセットと除外手順**:
  - テスト課金をした管理者・テスターのアカウントから課金形跡を除去し、テスト可能な非加入状態にリセットする際は、必ず以下の3点を行うこと：
    1. **本番/テストデータベースの課金ログ除外**: `payment_logs` テーブルから、対象 `user_id` の全ログデータを削除し、管理ダッシュボードの売上・DPU/MPU集計に影響を与えないようにする。
    2. **Stripeのアクティブな定期課金の解約**: 対象ユーザーのメールアドレスでStripeカスタマー（Stripe Dashboard または Stripe API）を検索し、アクティブ（`active` または `trialing`）なサブスクリプション契約が存在する場合は必ず明示的に解約（`cancel`）する。
    3. **データベースステータスのリセット**: 対象ユーザープロファイル（`user_profiles`）の `subscription_tier` を `'free'`、`subscription_status` を `'inactive'`、`has_used_trial` を `false` に更新し、次回のテストで「無料トライアル（7日間）」が正しく再起動する状態を担保する。

## 14. iOS Safariにおけるタップ領域ズレの防止と旧仕様ツアーの完全廃止に関する技術的制約 (v4.8)
- **backdrop-filter / backdrop-blur の完全排除規約**:
  - iOS Safari (WebKit) において、SPA遷移時（Next.js のクライアントサイド遷移）に `backdrop-filter: blur(...)` を持つ要素（モーダル、ヘッダー、進行中オーバーレイなど）が一度でもレンダリングされると、遷移後の画面で座標テストが狂い、タップ領域 of ズレやボタン of 無反応が発生する。
  - この致命的バグを防止するため、プロジェクト内のすべてのモーダル、ダイアログ、ローディング画面、およびヘッダー（`/quest` 画面、`/inn` 画面を問わない）において、**`backdrop-blur-*`（すりガラス効果）クラス of 記述は一切禁止**とする。
  - 代替として、透過度を最適化した高コントラスト半透明背景（`bg-black/85` や `bg-slate-950/95` など） of Pure CSS カラーのみを使用して背景を構成すること。
- **旧ツアー・推奨バッジロジック of 完全廃止**:
  - 新オンボーディング仕様では、施設説明ツアーは `onboardingTourStep`（1〜6）で一元管理され、ツアー終了後は全てのバッジが完全に消去される。
  - したがって、ツアー非アクティブ時（`isTourActive === false`） of 宿屋画面（`page.tsx`）において、過去 of `visited` フラグ（`visitedGuild`, `visitedAcademy` など）を用いて施設に推奨マークを付与する古い `recommendedFacility` 判定ロジックはバグを引き起こすため、完全に削除すること。非アクティブ時 of `recommendedFacility` は常に `null` とすること。
  - 同様に、第1話（6001）受注前 of 古い手動受注案内バナー（「ギルドで第1話...を受注しよう！」） of 表示ロジックも、新オンボーディングフローと重複するため、完全に削除すること。

## 15. クエストクリア後のアカウント認証プロモ判定時 of 誤タップ防止およびツアー開始遅延制御に関する技術的制約 (v4.9)
- **プロモーション表示判定中 of インタラクションブロック**:
  - クエスト帰還直後（`questJustCleared`）などでプロモモーダル（ゲスト登録プロモ、スターターパックプロモ） of 表示判定やロードが行われている間、背後 of 施設ボタンやヘッダーがタップできてしまうことを防ぐため、必ず `isPromoPending` のような判定中フラグを導入し、画面最前面に透明なガードカバー（`fixed inset-0 z-[300] bg-transparent pointer-events-auto`）を表示してすべての入力をブロックすること。
- **オンボーディングツアー開始 of 遅延（プロモ閉じるまでの保留）**:
  - プロモーション of 表示判定中または表示中である間は、ツアー of 自動開始（`onboardingTourStep = '1'`）を抑制すること。
  - プロモモーダルが閉じられた際、または表示不要と確定した段階で初めてツアーを開始（ステップ `'1'` に設定）させ、プロモとツアー of 不整合を防ぐこと。

## 16. プロモーション待機ガードの完全持続と遷移前 await によるフリーズ防止に関する技術的制約 (v4.9.1)
- **プロモ表示中における操作ガードカバーの持続と z-index 設計**:
  - プロモモーダルが表示されている（または表示準備中）の間は、誤クリックを防ぐガードカバー（`z-[300]`）をアクティブに保ち、プロモモーダルが閉じられたハンドラ内で初めてガードを解除（`isPromoPending = false`）すること。
  - この間、プロモモーダル（`GuestRegisterPromoModal`, `StarterPackPromoModal`）自体の z-index を必ず **`z-[1000]`** に引き上げ、ガードカバー（`z-[300]`）より手前に表示させることで、モーダル内部のボタン操作を妨げずに背後の誤操作を防ぐ。
- **遷移前非同期処理の await の徹底**:
  - `QuestResultModal` の `onClose` のように、画面遷移（`router.push`）を行う直前で行うプロフィール等の再取得非同期処理（`fetchUserProfile()` 等）は、必ず **`await` を付与して完了を待つ** こと。
  - `await` を怠ると、Next.js のルーター遷移処理が即座にメインスレッドをブロックし、ボタンの「読み込み中…」などの React ステート更新がブラウザにペイントされる前に画面がフリーズしてしまい、ボタンが変わらない不具合を引き起こす。
- **リザルトローディングオーバーレイの前面化**:
  - クエスト結果画面等における遷移中のローディングオーバーレイ（`isClosing && (...)`）は、z-index を **`z-[2000]`** に引き上げ、カードや背面にあるシナリオエンジンの古いスピナー（「画面を切り替え中...」）などを完全に覆い隠し、不細工な透過残像をシャットアウトすること。







## 17. ツアー自動開始のレースコンディション防止およびState駆動の描画保証付き遷移に関する技術的制約 (v4.9.2)
- **ツアー自動開始の二重ガードとハイドレーション修正**:
  - 宿屋遷移時のツアー自動開始判定において、プロモ判定の遅延や isPromoPending の初期化ラグによるツアー誤開始（ステップ '1' への初期化）を防ぐため、ツアー開始判定内で sessionStorage.getItem('wirth_dawn_quest_just_cleared') === 'true' も直接チェックし、プロモーション保留期間中の自動開始を完全にブロックすること。
  - isPromoPending などの sessionStorage 依存のステートは、ハイドレーションミスマッチを防止するため初期値を false とし、マウント時の useEffect で安全にクライアント側の値と同期させること。
- **State駆動の描画保証付き遷移 (Deferred Transition)**:
  - 遷移（router.push）が発生するモーダルのクローズ処理において、ボタンの「読み込み中…」や全体オーバーレイの描画（Paint）を遷移前に確実に完了させるため、クローズイベントハンドラで直接 onClose を呼ばず、isClosing === true を検知する useEffect 内で setTimeout (150ms 程度) を挟んでから onClose を実行する設計を徹底すること。これにより、ブラウザの描画を確実に画面にピクセル反映させてから遷移処理にスレッドを明け渡す。
