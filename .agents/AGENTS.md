# Wirth-Dawn 開発・運用ルール (KI)

## 1. メンテナンスモードおよび Edge Middleware に関する技術的制約
- **Edge Middleware における JWT デコードの注意点**:
  - Edge Runtime (Next.js Middleware 等) では、Base64Url のデコードを行う `atob` がパディング `=` を欠いている場合に `DOMException` をスローするため、必ず事前に `while (base64.length % 4) { base64 += '='; }` などのパディング補正処理を挟むこと。
  - Supabase SSR が書き込むクッキー（`sb-*-auth-token`）はサイズが大きい場合に自動で分割保存（`*.0`, `*.1` 等）され、かつ値全体が Base64 エンコードされている場合がある。そのため、`auth-token` を名前に含むすべてのクッキーを収集・ソート結合した上で、JSON パースの前に一度 Base64 デコード処理を行うロジックを維持すること。
- **サーバーコンポーネント内でのイベントハンドラ厳禁**:
  - Next.js のサーバーコンポーネント（特に `/maintenance` などの独立画面）では、HTML 要素に `onMouseOver` や `onMouseOut` などのイベントハンドラを直接渡してはならない。これを行うと、サーバーレンダリング時にクラッシュ（500エラー）を引き起こす。
  - 代替として、`<style>` タグや Pure CSS (`:hover` クラス) を使用してスタイリングのインタラクションを実現すること。
- **タイムゾーンの明示的指定**:
  - メンテナンス画面などの日付表示では、サーバーが UTC タイムゾーンで動作している場合があるため、`new Date()` の `getHours()` を直接呼ぶとずれた時刻が表示される。必ず日本時間 (JST: UTC+9) のオフセットを明示的に加算し、`getUTC*` 系のメソッドを用いてフォーマットすること。

## 2. リリース検証およびメンテナンス手順
- **事前アカウント検証の徹底**:
  - メンテナンス（および緊急リリース）のデプロイ時には、必ず事前に登録されているホワイトリストアカウント（きたむ様アカウント: `c1cf67dd-527a-497e-bf88-ce10c2cb516f`）での動作確認を徹底すること。
  - **次回以降のメンテナンス検証時、必ず「きたむ様アカウント」で実際にログインし、正常にメンテナンスをバイパスしてゲームに入れることを本番環境で徹底検証する。**
  - **Googleなどの OAuth 認証が行われる前段階（認証クッキーが存在しない状態）**では自動バイパスが機能しない。そのため、以下の2ステップで検証を行うこと：
    1. **未認証状態**: 管理者用バイパス URL（`?bypass=<bypass-key>`）を経由させてブラウザにバイパス用クッキーを植え付け、メンテナンス画面を通過してログイン画面が表示されるか確認する。
    2. **認証完了状態**: 実際に「きたむ様アカウント」でGoogleログインを行い、メンテナンス制限に引っかからずに拠点（宿屋等）に入れるか、一般ユーザーが遮断されている状態で正常にゲームをプレイできるか確認する。
- **環境間の逆マージ (バックポート) の確認**:
  - 本番環境（`main`）に直接適用されたマイグレーションやソースコードがある場合は、放置せず速やかに開発環境（`develop`）および PREVIEW DB へ逆マージし、環境間の不整合を防ぐこと。
